    Workflow Process - Data Braider Development¶

    Purpose: Guide AI agents to follow our team development workflow and quality gates.

    🏗️ DEVELOPMENT LIFECYCLE WORKFLOW¶

    When implementing features or fixing bugs in Data Braider, follow this structured workflow:

    PHASE 1: REQUIREMENTS CLARIFICATION¶

    BEFORE writing any code, clarify requirements with stakeholders:

    1. Identify Gaps in Requirements
    2. What is unclear or underspecified?
    3. What assumptions would you need to make?

    4. What edge cases haven't been addressed?

    5. Ask Clarifying Questions via Azure DevOps

    6. Post questions as comments on the work item
    7. Be specific about what you need to know

    8. Explain why the information is needed

    9. Document Decisions

    10. Record stakeholder answers in work item comments
    11. Update work item description with clarified requirements
    12. Ensure traceability from requirement to implementation

    DO NOT proceed to implementation until key questions are answered.

    PHASE 2: IMPLEMENTATION PLANNING¶

    Once requirements are clear:

    1. Create Implementation Plan
    2. Document in .aidocs/plans/ folder
    3. Use template from .aidocs/templates/

    4. Include:

      • Affected components
      • Data model changes
      • Integration points
      • Testing approach

    5. Consider Quality Requirements

    6. Error handling strategy
    7. Logging and diagnostics
    8. Performance implications

    9. Security considerations

    10. Plan Testing

    11. What needs to be tested?
    12. What test data is required?
    13. Manual vs. automated testing

    Get approval on implementation plan before coding.

    PHASE 3: IMPLEMENTATION¶

    Follow AL development best practices:

    1. Code with Quality
    2. Include XML documentation comments
    3. Handle errors appropriately
    4. Follow Data Braider naming conventions (SPBD prefix)

    5. Write clean, readable code

    6. Implement Logging

    7. Use event-based logging patterns
    8. Log important operations

    9. Include diagnostic information

    10. Write Tests (when applicable)

    11. Unit tests for business logic
    12. Integration tests for workflows
    13. Test error conditions

    PHASE 4: DOCUMENTATION¶

    After implementation is complete:

    1. Update Wiki Documentation
    2. Add to Setup section for configuration
    3. Add to Usage section for how-to guides

    4. Add to Support section for troubleshooting

    5. Maintain Consistency

    6. Match existing wiki style
    7. Use practical examples

    8. Include screenshots/diagrams when helpful

    9. Update .aidocs Index

    10. Link implementation plan
    11. Document key decisions
    12. Record completion status

    🎯 BEHAVIORAL RULES FOR AI AGENTS¶

    ALWAYS DO:¶

    ✅ Ask clarifying questions before making assumptions
    ✅ Document decisions in Azure DevOps
    ✅ Create implementation plans for non-trivial work
    ✅ Include error handling in all code
    ✅ Write tests for new functionality
    ✅ Update documentation after implementation
    ✅ Follow existing code patterns in the project

    NEVER DO:¶

    ❌ Start coding without clarifying vague requirements
    ❌ Make major assumptions without stakeholder input
    ❌ Skip documentation updates
    ❌ Ignore testing requirements
    ❌ Provide code without error handling
    ❌ Break existing functionality

    📋 WORK ITEM INTEGRATION¶

    Work Item States (Simplified)¶

    • To Do: Not started, requirements may need clarification
    • Doing: Active development in progress
    • Done: Implementation complete, tested, documented

    When Moving to "Doing"¶

    • Requirements are clear
    • Implementation plan exists
    • You're ready to code

    When Moving to "Done"¶

    • Code is complete and tested
    • Documentation is updated
    • Work item has traceability to implementation

    🧪 TESTING EXPECTATIONS¶

    For New Features¶

    • Write tests for business logic
    • Test error conditions and edge cases
    • Verify integration points with existing features

    For Bug Fixes¶

    • Create test that reproduces the bug
    • Verify fix resolves the issue
    • Test for regressions in related areas

    Test Data¶

    • Use realistic test scenarios
    • Consider BC demo data (CRONUS)
    • Document any special test setup required

    📚 DOCUMENTATION EXPECTATIONS¶

    Wiki Updates Required¶

    When implementing features that users will interact with:

    1. Setup Documentation
    2. How to configure the feature
    3. Prerequisites and permissions

    4. Initial setup steps

    5. Usage Documentation

    6. How to use the feature
    7. Common workflows

    8. Examples

    9. Support Documentation

    10. Troubleshooting common issues
    11. Error messages and solutions
    12. Performance tips

    Documentation Style¶

    • Practical and actionable - focus on how-to
    • Use examples - show, don't just tell
    • Scannable format - use tables, lists, headers
    • ADO wiki compatible - use :::mermaid for diagrams

    🔄 EXAMPLE WORKFLOW¶

    Scenario: Work item says "Add logging support"

    ❌ WRONG APPROACH (Vibe Coding)¶

    1. Look at work item
    2. Implement logging based on assumptions
    3. Commit code
    4. Consider it done

    ✅ CORRECT APPROACH (Knowledge-First)¶

    1. Review work item - identify vague areas
    2. Ask questions in ADO comments:
    3. "What should be logged? All API calls or only errors?"
    4. "How long should logs be retained?"
    5. "Who needs access to view logs?"
    6. "What performance impact is acceptable?"
    7. Wait for answers from stakeholders
    8. Create implementation plan based on clarified requirements
    9. Implement with proper logging, error handling, tests
    10. Update wiki with logging setup, usage, and troubleshooting guides
    11. Move to Done with full traceability
